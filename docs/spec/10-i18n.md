# 10 — Internationalization

> English and Korean, via react-i18next.

The UI is localized into English (`en`) and Korean (`ko`). The
project's primary user is Korean; English is the default for everyone
else and for source-code identifiers, commit messages, and docs (see
[`../decisions.md`](../decisions.md) D-10).

---

## Locale Selection

On first launch:

1. Read `winmux.toml`'s `[general].language`.
2. If `"system"` (default), detect via Tauri's `LocaleResolver`:
   - Reads Windows current language (typically `ko-KR` or `en-US`).
   - Normalizes to `ko` or `en`. Anything else falls back to `en`.
3. User can override in Settings → General.

The chosen language is persisted to `winmux.toml`.

---

## Library: react-i18next

```typescript
// src/lib/i18n.ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import en from '@/locales/en.json';
import ko from '@/locales/ko.json';

await i18n
  .use(initReactI18next)
  .init({
    resources: { en: { translation: en }, ko: { translation: ko } },
    lng: detectLocale(),
    fallbackLng: 'en',
    interpolation: { escapeValue: false },
    returnNull: false,
    keySeparator: '.',
    nsSeparator: false,
  });

export default i18n;
```

`returnNull: false` means missing keys return the key itself (and we
log a warning in dev). Better than crashing.

## Usage in Components

```tsx
import { useTranslation } from 'react-i18next';

export function NewSessionModal() {
  const { t } = useTranslation();
  return (
    <Dialog>
      <DialogTitle>{t('session.new.title')}</DialogTitle>
      <Button>{t('session.new.create')}</Button>
    </Dialog>
  );
}
```

For dynamic language switching:

```tsx
import i18n from '@/lib/i18n';

function LanguageSelect() {
  return (
    <select onChange={(e) => i18n.changeLanguage(e.target.value)}>
      <option value="en">English</option>
      <option value="ko">한국어</option>
    </select>
  );
}
```

## Translation Key Conventions

- **Keys are English** (`session.new.title`).
- **Values are localized strings.**
- **Hierarchical:** dot-separated namespaces.
- **Verb-noun form** for actions: `session.create`, not
  `createSession`.

Top-level namespaces:

| Namespace | Purpose |
| --- | --- |
| `app` | Window titles, tray menu items |
| `session` | Session-related UI |
| `window` | Window UI (tabs, rename) |
| `pane` | Pane controls |
| `settings` | Settings UI |
| `errors` | User-facing error messages |
| `notifications` | Toasts and balloons |
| `confirm` | Confirmation dialog text |
| `keys` | Key-binding labels |
| `status` | Status bar fragments |
| `tray` | Tray menu items |

## Plurals and Interpolation

react-i18next has built-in plural and interpolation. We use them:

```json
{
  "session": {
    "count_one": "{{count}} session",
    "count_other": "{{count}} sessions"
  }
}
```

```tsx
t('session.count', { count: 3 });  // "3 sessions"
```

For Korean, which doesn't pluralize nouns by count, we use the
`other` form only:

```json
{
  "session": {
    "count_other": "세션 {{count}}개"
  }
}
```

react-i18next resolves the appropriate form per locale.

## Text Direction

Both `en` and `ko` are LTR. WinMux does not target RTL languages
in M1–M4.

## Date and Time Formatting

Use the browser's `Intl.DateTimeFormat`:

```typescript
new Intl.DateTimeFormat(i18n.language, {
  dateStyle: 'short',
  timeStyle: 'medium',
}).format(date);
```

Times in logs and audit records are stored as ISO 8601 UTC and
formatted on display.

## Source of Truth

`en` is the source of truth. All keys are defined in `en.json` first;
`ko.json` mirrors the structure. A CI check (`npm run i18n:check`)
ensures:

- Every key in `en.json` exists in `ko.json`.
- Every key in `ko.json` exists in `en.json`.
- No empty values.
- Interpolation placeholders match between languages.

The script is in `scripts/check-i18n.mjs`.

---

## Missing Keys at Runtime

In dev mode (`import.meta.env.DEV`):

- Missing keys render as `[MISSING: key.name]`.
- Logged at WARN in the browser console (via the logger wrapper).
- Lint catches most cases.

In production:

- Missing keys fall back to the key itself.
- Logged at WARN in the file log.

---

## Non-UI Strings

The following are **not** localized:

- Log messages.
- Error codes (`SESSION_NOT_FOUND`, etc.) — only the user-facing
  message wrapping the code is localized.
- CLI output (English only).
- Configuration field names.
- Source-code identifiers.

The audit log and trace logs are diagnostic; English keeps them
greppable.

---

## Adding a Locale Later

Not planned for M1–M4. If a Japanese or Chinese contributor wants to
add their language:

1. Copy `en.json` to `ja.json` or `zh.json`.
2. Translate values.
3. Add to `i18n.ts` resources map.
4. Update `language` enum in `winmux.toml` schema.
5. Update locale-detection mapping in `detectLocale()`.

CI check ensures completeness.

## CJK Rendering

CJK characters (Hangul, kanji, hanzi) must render correctly even
when the UI language is English. This is a font-stack concern, not
an i18n concern. See [`07-tray-and-gui.md`](07-tray-and-gui.md) and
[`../nonfunctional/accessibility.md`](../nonfunctional/accessibility.md)
for the font fallback chain.

---

## IME Input

The IME (Korean default IME, Microsoft IME for Japanese, etc.)
composes text inside the xterm.js textarea. The composition is held
back from `PtyInput` until `compositionend`. See
[`04-key-handling.md`](04-key-handling.md) for the flow.
